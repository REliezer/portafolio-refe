content:
  description: |
    🛠️ This project was not only about building from scratch, but also about improving and expanding existing functionality.
    Throughout development, several technical challenges were addressed involving the Frontend, Backend, the asynchronous worker in Azure Functions, and the cloud infrastructure.
    Each improvement had the main goal of enhancing the user experience, improving system performance, and optimizing the handling of data coming from the PokeAPI.
  item:
  - nombre: "🗑️ Implement Full Report Deletion"
    resume: |
      The provided Report Generator system only allowed creating new reports based on the selected Pokémon type, so the first task was to let the user delete reports. To achieve this, the following Frontend and Backend changes were implemented:
    modulos:
    - An “Delete” button was added to each row of the reports table shown in the UI.
    - A modal dialog was implemented that appears when the user clicks the button, asking for confirmation before proceeding.
    - Creation and implementation of a new endpoint that allowed deleting the database record and the corresponding CSV file (report).
    - Handling of potential errors that could occur when deleting both the record and the file.
    - UI/UX improvements, such as updating the table and showing a confirmation message when deletion completed successfully.
  - nombre: "📊 Enrich the Report with Pokémon Details"
    resume: |
      The reports generated by the system only contained two fields: the Pokémon name (name) and the Pokémon URL (url). Therefore, the process needed to be modified to include more information using the existing URL.
      For this, the following changes were implemented in the worker (Azure Function), which was responsible for fetching data and generating the CSV files:
    modulos:
    - After obtaining the initial Pokémon list with each one’s name and URL, the worker iterated over each Pokémon using the URL to retrieve additional data such as height, weight, sprite, generation, types, stats (hp, attack, defense, special-attack, special-defense, speed), and abilities.
    - To keep the code cleaner and more organized, two new helper functions were created get_pokemon_info and get_pokemon_generation. The first returned all new fields in a list, which were then added to the existing ones. The second was implemented to obtain the Pokémon’s generation, since this value was not available directly from the initial URL and required a third request.
    - Handling of potential errors that could occur when making the requests.
  - nombre: "🎲 Reports with Random Sampling"
    resume: |
      The provided system was designed to generate reports using all Pokémon that belong to a given category (type), so a new option was requested where the user could enter a quantity (sample_size) representing the total number of records the report would contain. In addition, the returned records had to be random.
      For this task, changes were made to the Frontend, Backend, the worker (Azure Function), and the Database:
    modulos:
    - A numeric input field was added to the UI, where the user would enter the number of records to return.
    - In the database, a 'sample_size' INT NULL field was added to the 'request' table. This field stores the number of records requested when provided.
    - On the backend side, the 'sample_size' field was added to the Pydantic model, and the 'insert_pokemon_request' endpoint was modified to receive the new value and send it to the worker.
    - Once 'sample_size' was received by the worker, it generated only the requested number of records ('sample_size') and did so randomly, using the full collection of records as the base.
    - Handling of potential errors that could occur when making the requests.
    - Validation of the user input to ensure it is a positive integer.
    - UI/UX improvements, such as showing messages when the entered value is invalid.
